# 一、Kafka 基本介绍

Kafka 是由 Apache 基金会维护的一款**分布式流处理平台**，最初由 LinkedIn 开发并开源。其核心定位是“高吞吐量的分布式发布-订阅消息队列”，不仅能实现传统消息队列的“解耦、削峰、异步”功能，还能支撑实时流数据处理场景，是大数据生态中消息流转的关键组件。

核心设计理念：以“日志”为底层存储结构，通过分布式集群架构实现高并发、高可靠的数据传输，同时支持数据的持久化存储和回溯消费。

# 二、核心特性

- **高吞吐量**：通过批量发送、顺序读写磁盘、零复制等机制，单节点可支撑每秒数十万条消息的处理，适合大规模数据传输场景。

- **分布式高可用**：支持集群部署，通过“副本机制”（Replication）实现数据冗余，单个节点故障不影响集群整体可用性，容错性强。

- **持久化存储**：消息会持久化到磁盘，支持按时间或大小设置保留策略，可回溯历史数据用于数据恢复或离线分析。

- **多订阅者支持**：同一主题（Topic）可被多个消费者组订阅，每个消费者组独立消费消息，满足多场景数据复用需求。

- **流处理能力**：集成 Kafka Streams 组件，可直接基于 Kafka 进行实时流数据处理（如过滤、聚合、关联），无需依赖第三方计算框架。

# 三、基本使用方法（核心流程）

## 1. 核心概念铺垫

- **Topic（主题）**：消息的分类容器，生产者发送消息到指定主题，消费者从主题订阅消息，是消息流转的核心载体。Topic在业务应用配置上**并非只能有一个**，可根据业务需求创建多个。对于多种不同业务类型的消息，可通过“一个业务类型对应一个Topic”的方式隔离，也可在同一Topic内通过消息标签（如Key）区分，再由消费者过滤消费。

- **Partition（分区）**：主题的细分单元，一个主题可分为多个分区，分区内消息按“偏移量（Offset）”顺序存储，是并行处理的基础。**注意事项**：1. 分区副本需分散在不同Broker节点，避免单节点故障导致副本全失；2. 分区数一旦创建通常不可减少（可新增），需提前规划；3. 分区数决定消费者组最大并行度，需匹配消费能力。

- **Consumer Group（消费者组）**：由一组消费同一类业务消息的消费者实例组成的逻辑分组，是Kafka实现负载均衡和故障转移的核心机制。与Topic的关系：同一Topic的消息可被多个消费者组同时订阅（各组独立消费，互不干扰）；同一消费者组内的消费者实例可共同分摊消费**一个或多个Topic**的所有分区，且一个分区仅能被组内一个实例消费。**关键注意事项**：同一消费者组内的所有消费者实例，订阅的Topic列表必须完全相同，否则会导致重平衡异常或部分Topic消费缺失。配置方式：通过客户端代码指定消费者组ID（如Java的`group.id`、Golang sarama的`NewConsumerGroup`参数），同一组内所有实例必须配置相同的组ID；订阅多个Topic时，在客户端订阅接口中传入多个Topic名称即可（如Java的`subscribe(Arrays.asList("topic1", "topic2"))`）；可通过集群配置（如`group.min.session.timeout.ms`）调整组的会话超时等参数。

- **Producer（生产者）**：负责向 Kafka 集群发送消息的角色，可指定消息发送到的主题和分区。

- **Consumer（消费者）**：从 Kafka 集群订阅并消费消息的角色，本质是消费消息的应用进程/实例（非特指服务器）。消费者需归属一个“消费者组”，组内消费者共同分摊消费分区，且一个分区仅能被组内一个消费者消费。消费者实例可部署在同一台服务器或多台服务器上。

- **Broker**：Kafka 集群中的单个服务器节点，是集群的基础组成单元，主要负责存储消息（对应主题的分区及副本数据）和处理客户端（生产者/消费者）的读写请求。Broker的分布式部署是Kafka高可用的基础——分区的多个副本会分散存储在不同Broker上，当某个Broker故障时，其存储的副本若为跟随者（Follower），则不影响服务；若为首领（Leader），集群会从其他Broker的副本中选举新Leader，保障服务连续性。一个Kafka集群由多个Broker组成，通过Broker间的协同实现数据冗余和故障转移。

## 2. 基础使用步骤

1. **集群部署与启动**：搭建 Kafka 集群（单节点用于测试，生产环境至少 3 节点保证高可用），启动 Zookeeper（Kafka 旧版本依赖，新版本可使用 KRaft 模式替代）和 Kafka 服务。

2. **创建主题**：通过命令行或客户端工具创建主题，指定分区数和副本数。实际业务中可根据消息类型拆分主题（如订单消息主题order-topic、物流消息主题logistics-topic），示例：创建3分区、2副本的订单主题order-topic。

3. **生产者发送消息**：通过 Java、Python 等语言的 Kafka 客户端，编写生产者代码，向指定主题发送消息（支持同步/异步发送）。

4. **消费者消费消息**：编写消费者代码，核心是配置消费者组ID（如`group.id=order-log-group`），指定订阅的一个或多个Topic（如同时订阅`order-topic`和`log-topic`）；组内实例会自动分摊消费所有订阅Topic的分区，从集群拉取并处理消息（可设置消费起始偏移量，如从最新消息或最早消息开始消费）。

## Kafka 核心优势总结

1. 吞吐量碾压多数竞品，是大数据场景（如日志采集、用户行为追踪）的首选；2. 分布式架构容错性强，集群扩展方便；3. 数据持久化和回溯能力，兼顾消息队列和数据存储需求；4. 与流处理框架深度集成，可构建端到端的实时数据管道。

# 五、典型应用场景

1. **日志聚合**：收集分布式系统中各节点的日志数据，统一发送到 Kafka 集群，再由下游系统（如 ELK 栈）进行存储和分析。

2. **实时数据管道**：作为业务系统与大数据平台之间的桥梁，将业务数据实时同步到数据仓库、湖仓等，支撑实时报表和分析。

3. **微服务解耦**：在微服务架构中，通过 Kafka 实现服务间异步通信，解耦服务依赖，提升系统容错性（如订单系统向库存系统发送消息）。

4. **实时流处理**：基于 Kafka Streams 或集成 Flink/Spark Streaming，处理实时数据流（如实时推荐、实时风控、监控告警）。

5. **消息通知系统**：支撑高并发的消息推送场景（如电商促销短信、APP 推送），通过高吞吐量应对峰值流量。

# 六、必须了解的重要知识点

- **分区机制**：分区是 Kafka 并行处理的核心，一个分区只能被消费者组内的一个消费者实例（非特指服务器）消费，因此分区数决定了消费者组的最大并行消费能力。**注意事项**：1. 分区数需结合集群Broker数量规划，确保副本均匀分布；2. 过多分区会增加集群元数据管理开销，需避免盲目扩容；3. 消息按Key路由分区时，需确保Key分布均匀，避免分区负载失衡。消费者实例可部署在单台或多台服务器，核心是实例数量与分区数匹配以实现并行。

- **偏移量（Offset）**：每个分区内的消息都有唯一偏移量，消费者通过记录偏移量实现“断点续传”，需注意偏移量的提交机制（自动提交/手动提交）。

- **副本与首领节点**：每个分区有多个副本，其中一个为“首领节点”（Leader），负责处理读写请求；其余为“跟随者节点”（Follower），同步 Leader 数据，Leader 故障时自动切换。

- **消息确认机制（ACK）**：生产者可设置 ACK 级别（0：不确认；1：Leader 写入成功确认；-1：所有副本写入成功确认），平衡吞吐量与可靠性。

- **消费者组重平衡**：当消费者组内消费者数量变化、订阅的Topic分区变化或实例心跳超时（默认10秒）时，会触发重平衡，重新分配所有订阅Topic的分区与组内消费者的对应关系，期间会导致短时间消费暂停。**关键注意事项**：同一消费者组内所有实例订阅的Topic必须完全一致，否则重平衡时无法正确分配分区；重平衡是消费者组实现负载均衡的核心机制，需合理配置会话超时参数避免频繁触发；多Topic消费时，需确保组内实例数量匹配所有订阅Topic的总分区数，避免负载失衡。

- **KRaft 模式**：Kafka 2.8.0 后推出的无 Zookeeper 模式，通过 Kafka 自身的 Raft 协议管理集群元数据，简化部署和维护。
