# 一、MVCC 是什么？

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 MySQL 中 InnoDB 存储引擎实现事务隔离级别的核心技术之一。其核心思想是：**为数据库中的每条记录保存多个版本**，当并发操作时，不同事务可访问对应版本的记录，避免直接加锁导致的性能损耗，实现“读不加锁、写不阻塞读”的高效并发控制。

关键前提：仅 InnoDB 支持 MVCC，MyISAM 等其他引擎不支持（因不支持事务）。

# 二、MVCC 的核心实现原理

InnoDB 的 MVCC 依赖三大核心组件实现：**隐藏字段**、**undo log（回滚日志）**、**Read View（读视图）**。三者协同工作，确保不同事务看到对应版本的记录。

## 1. 隐藏字段：记录版本基础信息

InnoDB 会为表中的每条记录自动添加 3 个隐藏字段（无需手动定义），用于标记记录的版本和事务关联信息：

- **DB_TRX_ID（事务ID）**：最近一次修改该记录的事务 ID（6字节）。无论是插入、更新还是删除，都会记录操作的事务 ID。

- **DB_ROLL_PTR（回滚指针）**：指向该记录的上一个版本在 undo log 中的地址（7字节），形成一条“版本链”。

- **DB_ROW_ID（行ID）**：若表未指定主键且无唯一索引，InnoDB 会自动生成该字段作为默认主键（6字节），与版本控制直接关联度低。

## 2. undo log：存储历史版本数据

undo log（回滚日志）是事务执行过程中，用于记录“数据修改前状态”的日志文件，核心作用有二：**事务回滚**（出错时恢复数据）、**构建版本链**（供 MVCC 访问历史版本）。

当对记录执行修改操作时，InnoDB 会先将修改前的记录完整备份到 undo log 中，再修改原记录。同时，通过 DB_ROLL_PTR 字段将原记录与 undo log 中的历史版本关联，形成“版本链”。

示例：一条记录被3个事务依次更新，版本链结构如下：

当前记录（DB_TRX_ID=3, DB_ROLL_PTR→undo log 版本2） → undo log 版本2（DB_TRX_ID=2, DB_ROLL_PTR→undo log 版本1） → undo log 版本1（DB_TRX_ID=1, DB_ROLL_PTR=NULL）

## 3. Read View：决定事务能看到哪个版本

Read View（读视图）是事务开启时生成的一个“快照”，用于定义当前事务能访问的“可见版本范围”。其核心是维护了4个关键参数，通过对比记录版本的 DB_TRX_ID 来判断可见性：

- **up_limit_id**：当前所有活跃事务中，最小的事务 ID（未提交的事务）。

- **low_limit_id**：下一个即将生成的事务 ID（确保新事务 ID 唯一递增）。

- **trx_ids**：当前所有活跃事务的 ID 集合。

- **creator_trx_id**：生成该 Read View 的事务 ID（即当前事务 ID）。

**可见性判断规则**：对于版本链中的某条记录版本（其 DB_TRX_ID 为 trx_id）：

1. 若 trx_id = creator_trx_id：当前事务修改的记录，可见。

2. 若 trx_id &lt; up_limit_id：修改该记录的事务已提交，可见。

3. 若 trx_id > low_limit_id：修改该记录的事务是后续新事务，不可见。

4. 若 up_limit_id ≤ trx_id ≤ low_limit_id：查看 trx_id 是否在 trx_ids 中。不在则已提交（可见），在则未提交（不可见）。

关键差异：**读已提交（RC）**每次查询都会重新生成 Read View；**可重复读（RR）**仅在事务开启时生成一次 Read View（这也是 RR 隔离级别实现“可重复读”的核心原因）。

# 三、MVCC 解决了什么核心问题？

MVCC 的设计直接针对并发场景中的“读-写”“写-读”冲突问题，结合事务隔离级别，解决了以下核心痛点：

## 1. 解决“读不加锁、写不阻塞读”的并发效率问题

传统的“悲观锁”机制中，读操作会加共享锁，写操作会加排他锁，导致“读阻塞写、写阻塞读”，并发效率极低。而 MVCC 通过版本链和 Read View，让读操作直接访问历史版本，无需加锁；写操作仅修改当前版本，不影响读操作访问历史版本，极大提升了高并发场景下的吞吐量。

## 2. 实现不同事务隔离级别（核心支撑 RR 和 RC）

MySQL 的事务隔离级别中，**读已提交（RC）**和**可重复读（RR）**完全依赖 MVCC 实现：

- **读已提交（RC）**：每次查询重新生成 Read View，确保只能看到已提交的记录版本，避免“脏读”。

- **可重复读（RR）**：仅事务开启时生成一次 Read View，后续查询复用该视图，确保同一事务内多次查询结果一致，避免“不可重复读”和“幻读”（配合间隙锁解决幻读）。

## 3. 避免“脏读”“不可重复读”问题

- **杜绝脏读**：未提交事务的修改会记录在版本链中，但由于其事务 ID 在 trx_ids 中（活跃事务），其他事务的 Read View 会判断该版本不可见，避免读取未提交的“脏数据”。

- **解决不可重复读**：RR 隔离级别下，事务内复用 Read View，即使其他事务修改并提交了记录，当前事务仍访问开启时的版本，确保多次读取结果一致。

## 4. 降低锁竞争开销

MVCC 减少了悲观锁的使用场景，仅在“写-写”冲突时才需要加行锁（排他锁），而“读-写”“写-读”冲突通过版本控制解决，大幅降低了锁竞争导致的上下文切换和等待开销，提升了数据库的并发处理能力。

# 四、核心总结

|核心组件|作用|
|---|---|
|隐藏字段|标记记录的修改事务ID和历史版本指针|
|undo log|存储历史版本，构建版本链|
|Read View|定义可见版本范围，实现隔离级别|
MVCC 本质是通过“多版本快照”替代“直接锁竞争”，在保证事务隔离性的同时，最大化提升并发效率，是 InnoDB 成为 MySQL 默认存储引擎的核心原因之一。

# 五、典型场景执行流程

## 1. 读场景执行流程（以 RR 隔离级别为例）

读操作（SELECT）通过 MVCC 访问历史版本，全程无锁，具体步骤如下：

1. **生成读视图**：事务首次执行读操作时（RR 级别仅生成一次），InnoDB 生成 Read View，记录当前活跃事务 ID 集合、最小活跃事务 ID（up_limit_id）等参数。

2. **定位目标记录**：根据查询条件找到聚簇索引中的当前记录，获取该记录的 DB_TRX_ID（最近修改事务 ID）和 DB_ROLL_PTR（回滚指针）。

3. **判断版本可见性**：用 Read View 的规则校验当前记录的 DB_TRX_ID：
- 若可见，直接返回该记录数据；
- 若不可见，通过 DB_ROLL_PTR 回溯到 undo log 中的上一个历史版本，重复校验直到找到可见版本或版本链结束（无可见版本则返回空）。

RC 隔离级别与 RR 的差异：每次执行 SELECT 都会重新生成 Read View，因此能看到查询瞬间已提交的最新版本。

## 2. 写场景执行流程（以 UPDATE 为例）

写操作（UPDATE/DELETE/INSERT）需保证数据一致性，会加行锁并维护版本链，且**所有字段判断（如UPDATE WHERE）均基于记录最新版本**，步骤如下：

1. **加行锁并判断条件**：事务对目标记录发起写操作时，先尝试加排他行锁（防止其他事务同时修改），若锁被占用则阻塞等待；获取锁后，基于当前最新记录的字段值校验WHERE条件，不满足则直接释放锁结束操作。

2. **记录 undo log**：将修改前的记录完整备份到 undo log 中，生成一条 undo 日志记录。

3. **修改记录并更新隐藏字段**：
- 直接修改原记录的数据内容；
- 将该事务的 ID 赋值给记录的 DB_TRX_ID；
- 将 undo log 中历史版本的地址赋值给 DB_ROLL_PTR，形成新的版本链。

4. **提交事务**：事务提交后，释放行锁，undo log 保留（供其他事务读和自身回滚，后续由 purge 线程清理过期版本）。

## 3. 写并发场景执行流程

当多个事务同时修改同一条记录时，MVCC 结合行锁实现并发控制，流程如下（以事务 A 和事务 B 并发 UPDATE 为例）：

1. **事务 A 先获取锁**：事务 A 率先找到目标记录，成功加排他行锁，执行写流程：备份 undo log → 修改记录 → 更新 DB_TRX_ID 和 DB_ROLL_PTR，此时事务 A 未提交。

2. **事务 B 阻塞等待锁**：事务 B 发起对同一记录的 UPDATE，尝试加排他行锁时发现已被事务 A 占用，立即进入阻塞队列等待锁释放。

3. **事务 A 提交释放锁**：事务 A 执行 COMMIT，释放排他行锁，InnoDB 唤醒阻塞队列中的事务 B。

4. **事务 B 获取锁执行写操作**：事务 B 获得排他行锁，将事务 A 修改后的记录作为“修改前版本”备份到 undo log，然后修改记录内容，更新 DB_TRX_ID（为事务 B 的 ID）和 DB_ROLL_PTR（指向事务 A 修改后的版本在 undo log 的地址），最后提交事务并释放锁。

**关键说明**：写并发时通过“排他行锁”保证同一时间仅一个事务能修改记录，通过 undo log 维护多版本链，确保未获取锁的事务可通过读流程访问历史版本（不阻塞读）。
