MySQL 锁机制

一、锁分类

graph TD
    Lock[MySQL 锁] --> Level[按粒度]
    Lock --> Type[按类型]
    
    Level --> Global[全局锁]
    Level --> Table[表级锁]
    Level --> Row[行级锁]
    Level --> Page[页级锁]
    
    Type --> Read[读锁/共享锁 S]
    Type --> Write[写锁/排他锁 X]

二、全局锁

-- 加全局读锁（整库只读）
FLUSH TABLES WITH READ LOCK;

-- 释放锁
UNLOCK TABLES;

-- 使用场景：全库逻辑备份
-- 缺点：业务完全停滞

三、表级锁

MyISAM 表锁

-- 加读锁（其他会话可读，不可写）
LOCK TABLES my_table READ;

-- 加写锁（独占）
LOCK TABLES my_table WRITE;

-- 解锁
UNLOCK TABLES;

InnoDB 意向锁

-- 意向共享锁（IS）：事务准备给某些行加 S 锁
-- 意向排他锁（IX）：事务准备给某些行加 X 锁

-- 兼容性矩阵
--       IS  IX  S   X
-- IS    Y   Y   Y   N
-- IX    Y   Y   N   N
-- S     Y   N   Y   N
-- X     N   N   N   N

四、行级锁（InnoDB）

-- 共享锁（S）
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;

-- 排他锁（X）
SELECT * FROM user WHERE id = 1 FOR UPDATE;

-- 更新自动加 X 锁
UPDATE user SET name = 'Alice' WHERE id = 1;

五、行锁算法

算法说明适用场景
Record Lock锁定单个记录等值查询（唯一索引）
Gap Lock锁定间隙，不包含记录本身范围查询，防止幻读
Next-Key LockRecord + Gap Lock默认行锁算法
-- 表中有记录 id=10, 20, 30
-- Next-Key Lock 锁定区间
-- (-∞, 10], (10, 20], (20, 30], (30, +∞)

SELECT * FROM user WHERE id > 15 FOR UPDATE;
-- 锁定 (10,20], (20,30], (30,+∞)

七、乐观锁 vs 悲观锁

特性悲观锁乐观锁
实现SELECT FOR UPDATE版本号/CAS
冲突处理先加锁提交时检测
适用写多读少读多写少
并发度低高
-- 乐观锁实现
UPDATE product 
SET stock = stock - 1, version = version + 1 
WHERE id = 1 AND version = 1;

八、锁优化建议

  • 尽量使用行锁而非表锁（InnoDB）
  • 索引让行锁更精确
  • 控制事务大小，减少锁持有时间
  • 按固定顺序访问资源，避免死锁
  • 使用乐观锁减少锁竞争
