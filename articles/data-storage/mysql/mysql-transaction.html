MySQL 事务与锁

一、事务 ACID

特性说明
Atomicity（原子性）要么全部成功，要么全部回滚
Consistency（一致性）事务前后数据完整性约束不被破坏
Isolation（隔离性）事务之间相互隔离
Durability（持久性）事务提交后永久保存

二、事务控制

-- 开启事务
START TRANSACTION;
-- 或
BEGIN;

-- DML 操作
INSERT INTO account VALUES (1, 1000);
UPDATE account SET balance = balance - 100 WHERE id = 1;

-- 提交或回滚
COMMIT;    -- 提交
ROLLBACK;  -- 回滚

-- 保存点
SAVEPOINT sp1;
ROLLBACK TO sp1;  -- 回滚到保存点

三、隔离级别

隔离级别脏读不可重复读幻读
READ UNCOMMITTED可能可能可能
READ COMMITTED不可能可能可能
REPEATABLE READ（默认）不可能不可能可能
SERIALIZABLE不可能不可能不可能
-- 设置隔离级别
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 查看当前隔离级别
SELECT @@transaction_isolation;

五、锁机制

锁类型

锁说明
共享锁（S）SELECT ... LOCK IN SHARE MODE
排他锁（X）SELECT ... FOR UPDATE / DML
意向共享锁（IS）表级，表示事务将获取行级 S 锁
意向排他锁（IX）表级，表示事务将获取行级 X 锁

锁粒度

-- 行锁（InnoDB 默认）
SELECT * FROM user WHERE id = 1 FOR UPDATE;

-- 表锁（MyISAM 或显式锁定）
LOCK TABLES user WRITE;
UNLOCK TABLES;

-- 间隙锁（Gap Lock）
-- 防止幻读，锁定范围但不锁定记录本身
SELECT * FROM user WHERE id > 5 AND id < 10 FOR UPDATE;

七、死锁处理

-- 查看死锁日志
SHOW ENGINE INNODB STATUS;

-- 死锁检测（默认开启）
innodb_deadlock_detect = ON

-- 死锁超时
innodb_lock_wait_timeout = 50

八、乐观锁 vs 悲观锁

特性悲观锁乐观锁
机制先获取锁提交时检测冲突
SQLSELECT FOR UPDATEUPDATE + version 检查
适用写多读少读多写少
冲突阻塞等待重试/报错

乐观锁实现

-- 添加版本号字段
ALTER TABLE product ADD COLUMN version INT DEFAULT 0;

-- 更新时检查版本
UPDATE product 
SET stock = stock - 1, version = version + 1 
WHERE id = 1 AND version = 1;

-- 影响行数为 0 表示冲突，需重试

九、最佳实践

  • 事务尽量短小，减少锁持有时间
  • 按固定顺序访问资源，避免死锁
  • 使用 READ COMMITTED 减少锁竞争
  • 热点数据考虑乐观锁或缓存
  • 监控 innodb_row_lock_waits 指标
