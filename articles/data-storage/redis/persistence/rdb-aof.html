Redis 持久化：RDB 与 AOF

一、为什么需要持久化

Redis 是内存数据库，数据存储在内存中，重启后数据会丢失。持久化机制用于将内存数据保存到磁盘，实现数据恢复和备份。

二、RDB（Redis Database）

原理

定时将内存数据快照保存到二进制文件（dump.rdb）。

graph LR
    A[内存数据] -->|bgsave| B[子进程]
    B --> C[生成 RDB 文件]
    C --> D[替换旧文件]

触发方式

# 自动触发（redis.conf）
save 900 1      # 900秒内至少1个key变化
save 300 10     # 300秒内至少10个key变化
save 60 10000   # 60秒内至少10000个key变化

# 手动触发
SAVE            # 阻塞主进程
BGSAVE          # 后台异步执行

优点

  • 文件紧凑，体积小
  • 恢复速度快（直接加载二进制）
  • 适合备份和灾难恢复

缺点

  • 可能丢失最后一次快照后的数据
  • 大数据量时 fork 子进程耗时

三、AOF（Append Only File）

原理

记录所有写操作命令，重启时重新执行这些命令恢复数据。

# AOF 文件内容示例
*2\r\n$6\r\nSELECT\r\n$1\r\n0\r\n
*3\r\n$3\r\nSET\r\n$3\r\nkey\r\n$5\r\nvalue\r\n

同步策略

# redis.conf
appendfsync always     # 每次写入都同步，最安全，最慢
appendfsync everysec   # 每秒同步，默认，平衡
appendfsync no         # 由操作系统决定，最快，最不安全

AOF 重写

AOF 文件会越来越大，需要定期重写压缩：

# 重写触发条件
auto-aof-rewrite-percentage 100  # 增长100%时重写
auto-aof-rewrite-min-size 64mb   # 最小64MB才重写

# 重写原理
# 1. 创建子进程
# 2. 子进程遍历内存数据，生成新的 AOF 文件
# 3. 重写期间的新命令写入缓冲区
# 4. 重写完成后追加缓冲区内容

优点

  • 数据更安全，最多丢失1秒数据
  • 可读性好（文本格式）
  • 支持手动修改恢复

缺点

  • 文件体积大
  • 恢复速度慢（需要重放命令）
  • 写性能略低于 RDB

四、RDB vs AOF 对比

维度RDBAOF
文件体积小（二进制压缩）大（文本命令）
恢复速度快慢
数据安全可能丢数据最多丢1秒
可读性不可读可读
性能影响fork 时短暂阻塞持续磁盘写入

五、混合持久化（Redis 4.0+）

结合 RDB 和 AOF 的优点：

  • AOF 文件前半部分是 RDB 格式的全量数据
  • 后半部分是 AOF 格式的增量数据 
    # 开启混合持久化
aof-use-rdb-preamble yes

# 重写后的 AOF 文件结构
# [RDB 格式数据][AOF 格式增量数据]

    六、最佳实践

    方案1：仅 RDB（数据可重建）

    save 900 1
save 300 10
save 60 10000
appendonly no

    方案2：RDB + AOF（生产推荐）

    save 900 1
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes

    方案3：仅 AOF（数据安全优先）

    appendonly yes
appendfsync always  # 性能较差

    七、恢复流程

    # 1. 优先加载 AOF（如果开启）
# 2. AOF 不存在或关闭，加载 RDB
# 3. 都加载失败，启动空实例

redis-server --appendonly yes  # 指定配置启动

    八、注意事项

    • 不要同时关闭 RDB 和 AOF（除非纯缓存场景）
    • 定期备份 RDB/AOF 文件到异地
    • 监控磁盘空间和写入延迟
    • 大内存实例注意 fork 的内存消耗（Copy-on-Write）
