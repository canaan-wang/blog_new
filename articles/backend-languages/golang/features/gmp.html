Go GMP 调度模型详解

GMP 是 Go 运行时（runtime）的协程调度模型，由三个核心组件构成：

  • G（Goroutine）：用户态轻量级线程
  • M（Machine）：操作系统线程
  • P（Processor）：逻辑处理器，G 和 M 的桥梁

核心组件详解

G - Goroutine

type g struct {
    stack       stack   // 栈内存（初始 2KB，可动态增长）
    sched       gobuf   // 调度上下文（PC、SP、寄存器）
    goid        int64   // 唯一 ID
    status      uint32  // 状态
    ...
}

Goroutine 特点：

  • 轻量级：初始栈仅 2KB（线程通常 1-8MB）
  • 动态栈：按需增长/收缩（最大 1GB）
  • 状态：_Gwaiting、_Grunnable、_Grunning、_Gdead 等

M - Machine

type m struct {
    g0      *g      // 调度栈（系统 goroutine）
    curg    *g      // 当前运行的用户 goroutine
    p       puintptr // 绑定的 P
    ...
}

M 特点：

  • 对应一个 OS 线程
  • 默认最多 10000 个（可调整）
  • 阻塞时会释放 P，让其他 M 接管

P - Processor

type p struct {
    runqhead uint32      // 本地队列头
    runqtail uint32      // 本地队列尾
    runq     [256]guintptr // 本地可运行队列
    ...
}

P 特点：

  • 数量默认等于 CPU 核心数（GOMAXPROCS）
  • 每个 P 有独立的本地队列（无锁访问）
  • 存放待运行的 G

调度策略

1. 任务窃取（Work Stealing）

当 P 的本地队列为空时，从其他 P 偷取 G：

sequenceDiagram
    participant M1 as M1+P1
    participant M2 as M2+P2
    participant GQ as Global Queue
    
    M1->>M1: 本地队列空
    M1->>M2: 尝试偷取一半 G
    alt 偷取成功
        M2-->>M1: 转移 G
        M1->>M1: 执行 G
    else 偷取失败
        M1->>GQ: 从全局队列获取
    end

2. 调度时机

  • 主动调度：G 执行 runtime.Gosched()
  • 系统调用：G 阻塞在系统调用，M 释放 P
  • 协作式调度：函数调用时检查是否需要切换
  • 抢占式调度：Go 1.14+，基于信号的抢占

3. 抢占式调度

Go 1.14 之前只有协作式调度，可能导致长循环不释放 CPU。1.14 引入基于信号的抢占：

// 协作式调度示例（需要函数调用才会让出）
func longLoop() {
    for i := 0; i < 1e9; i++ {
        // 纯计算，无函数调用
        // 在 Go 1.13 可能一直占用 CPU
    }
}

// 解决：手动插入调度点
func longLoop() {
    for i := 0; i < 1e9; i++ {
        if i % 1000 == 0 {
            runtime.Gosched() // 主动让出
        }
    }
}

// Go 1.14+ 自动抢占，无需手动处理

GOMAXPROCS

控制 P 的数量，即并行执行的 goroutine 数：

import "runtime"

// 获取当前 GOMAXPROCS
n := runtime.GOMAXPROCS(0)
fmt.Println("当前 P 数量:", n) // 默认等于 CPU 核心数

// 设置 GOMAXPROCS（一般不需要）
runtime.GOMAXPROCS(4)

调度器状态查看

# 打印调度器跟踪信息
GODEBUG=schedtrace=1000 ./program

# 输出示例：
# SCHED 0ms: gomaxprocs=8 idleprocs=5 threads=5 spinningthreads=1 idlethreads=2 runqueue=0 [0 0 0 0 0 0 0 0]

最佳实践

场景建议
CPU 密集型GOMAXPROCS = CPU 核心数（默认）
I/O 密集型可适当增加 GOMAXPROCS
长时间计算避免纯循环，定期让出或插入函数调用
大量 goroutine使用 worker pool 控制并发度

总结

组件作用数量
G用户任务动态（百万级）
M执行线程动态（默认上限 10000）
P调度上下文GOMAXPROCS（默认 CPU 核数）

GMP 模型通过：

  • 本地队列减少锁竞争
  • Work Stealing 平衡负载
  • 抢占式调度保证公平

实现了高效、轻量的 goroutine 调度，是 Go 高并发能力的核心支撑。
