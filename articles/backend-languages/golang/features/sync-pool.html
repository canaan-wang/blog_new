Go sync.Pool 详解

sync.Pool 是 Go 提供的对象池，用于缓存和复用临时对象，减少 GC 压力和内存分配开销，特别适合高并发场景下频繁创建和销毁对象的场景。

适用场景

  • 频繁分配和释放相同类型的对象（如 buffer、结构体）
  • 需要减少 GC 压力的高性能服务
  • 创建成本高的对象（如大数组、复杂结构体）

核心 API

方法说明
Get() any从池中获取对象（池空则调用 New 创建）
Put(x any)将对象归还池中

使用示例

基础使用

package main

import (
    "bytes"
    "fmt"
    "sync"
)

// 创建 bytes.Buffer 对象池
var bufferPool = sync.Pool{
    New: func() any {
        return new(bytes.Buffer)
    },
}

func processData(data string) string {
    // 从池中获取 buffer
    buf := bufferPool.Get().(*bytes.Buffer)
    defer bufferPool.Put(buf) // 使用完归还

    buf.Reset() // 重置后再使用
    buf.WriteString("Processed: ")
    buf.WriteString(data)
    
    return buf.String()
}

func main() {
    for i := 0; i < 10; i++ {
        result := processData(fmt.Sprintf("data-%d", i))
        fmt.Println(result)
    }
}

大缓冲区复用

var bigBufferPool = sync.Pool{
    New: func() any {
        return make([]byte, 64*1024) // 64KB 缓冲区
    },
}

func handleRequest(data []byte) []byte {
    buf := bigBufferPool.Get().([]byte)
    defer bigBufferPool.Put(buf)

    // 处理数据...
    copy(buf, data)
    
    // 返回处理结果（需要复制，因为 buf 要归还池）
    result := make([]byte, len(data))
    copy(result, buf[:len(data)])
    return result
}

底层原理

type Pool struct {
    noCopy noCopy

    local     unsafe.Pointer // [P]poolLocal 数组，每个 P 一个
    localSize uintptr        // local 数组大小

    victim     unsafe.Pointer // 上一轮 GC 的 local，用于缓冲
    victimSize uintptr

    New func() any // 创建新对象的函数
}

type poolLocal struct {
    private any      // 私有对象（仅当前 P 可用）
    shared  poolChain // 共享链表（当前 P 生产，其他 P 可偷取）
}

设计特点

graph TD
    A["Get 操作"] --> B{"private 有对象?"}
    B -->|是| C["直接返回 private"]
    B -->|否| D["从 shared 链表获取"]
    D --> E{"shared 有对象?"}
    E -->|否| F["从其他 P 的 shared 偷取"]
    F --> G{"偷取成功?"}
    G -->|否| H["调用 New 创建"]
    
    I["Put 操作"] --> J{"private 为空?"}
    J -->|是| K["放入 private"]
    J -->|否| L["放入 shared 链表"]
  • 每个 P 独立：每个处理器（P）有自己的 poolLocal，减少锁竞争
  • private 优先：无锁访问，性能最优
  • shared 共享：当前 P 生产的对象可被其他 P 偷取（work stealing）
  • victim 缓冲：GC 后保留上一轮的对象，减少 GC 后立即分配

使用注意事项

1. 对象归还前必须重置

type Buffer struct {
    data []byte
}

var pool = sync.Pool{
    New: func() any { return &Buffer{} },
}

func useBuffer() {
    buf := pool.Get().(*Buffer)
    defer pool.Put(buf)

    buf.data = buf.data[:0] // 重置！清除旧数据
    // 使用 buf...
}

2. 不要存储池中的对象

// 错误：保存了池中对象的引用
var globalBuf *bytes.Buffer

func bad() {
    buf := pool.Get().(*bytes.Buffer)
    globalBuf = buf // 危险！
    pool.Put(buf)   // 此时 buf 可能被其他 goroutine 使用
}

3. 不要对 Pool 做类型断言外的假设

// 错误假设：Get 总是返回非 nil
buf := pool.Get().(*bytes.Buffer) // 如果 New 为 nil，Get 返回 nil
if buf == nil {
    // 处理 nil 情况
}

4. 对象会被 GC 回收

// 注意：sync.Pool 中的对象可能在 GC 时被回收
// 不要假设 Put 的对象一定能再次 Get 到

var pool = sync.Pool{
    New: func() any {
        fmt.Println("Create new") // GC 后可能再次打印
        return &MyStruct{}
    },
}

5. 适合无状态对象

// 适合：无状态、可重置的对象
var bytePool = sync.Pool{
    New: func() any { return make([]byte, 1024) },
}

// 不适合：带状态、与特定请求绑定的对象（如 db 连接）
// db 连接应使用专门的连接池

性能对比

// 不使用 Pool：每次分配新内存
func withoutPool(n int) {
    for i := 0; i < n; i++ {
        buf := make([]byte, 1024)
        _ = buf
    }
}

// 使用 Pool：复用内存
func withPool(n int) {
    for i := 0; i < n; i++ {
        buf := pool.Get().([]byte)
        _ = buf
        pool.Put(buf)
    }
}

// 在高并发场景下，使用 Pool 可减少 90%+ 的内存分配

最佳实践

建议说明
用完即还使用 defer 确保 Put 被调用
重置状态归还前清除对象中的敏感数据
设置 New确保 Get 不会返回 nil
无状态对象只缓存无状态、可重置的对象
不要依赖对象可能在 GC 时被回收

总结

sync.Pool 是高并发场景下的性能优化利器，通过对象复用显著减少内存分配和 GC 压力。但要注意：

  • 仅用于无状态临时对象
  • 使用完必须归还
  • 归还前重置状态
  • 不要保存池中对象的引用
