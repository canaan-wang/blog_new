Go GC（垃圾回收）详解

Go 使用并发标记-清除（Concurrent Mark and Sweep）垃圾回收器，特点是低延迟、并发执行，最大程度减少 STW（Stop The World）时间。

GC 算法演进

版本算法特点
Go 1.0-1.2串行标记-清除STW 时间长，可达数百毫秒
Go 1.3并行清除清除阶段并发，减少 STW
Go 1.5并发标记引入三色标记，标记与业务并发
Go 1.8混合写屏障几乎消除 STW，通常 < 100μs

三色标记法

Go GC 使用三色标记法追踪存活对象：

graph TD
    W[白色: 未访问/垃圾] --> G[灰色: 访问中/待处理]
    G --> B[黑色: 已访问/存活]
    
    style W fill:#fff,stroke:#333
    style G fill:#ff9,stroke:#333
    style B fill:#9f9,stroke:#333
  • 白色：潜在的垃圾对象
  • 灰色：已被访问，但引用的对象还未全部扫描
  • 黑色：已被访问，且引用的对象都已扫描（确定存活）

标记流程

  1. 所有对象初始为白色
  2. 从根对象（全局变量、栈上变量）开始，标记为灰色
  3. 扫描灰色对象，将其引用的对象标记为灰色，自身标记为黑色
  4. 重复步骤 3 直到没有灰色对象
  5. 清除所有白色对象

写屏障（Write Barrier）

并发标记时，业务代码可能修改对象引用，导致漏标。Go 使用混合写屏障解决：

// 伪代码：写屏障逻辑
func writePointer(slot, ptr) {
    //  shade(ptr)      // 新引用标记为灰色（插入写屏障）
    //  shade(*slot)    // 旧引用标记为灰色（删除写屏障）
    *slot = ptr
}
  • 插入屏障：新引用对象标记为灰色
  • 删除屏障：旧引用对象标记为灰色
  • 混合屏障：Go 1.8+ 使用，兼顾两者优点

GC 触发时机

  1. 内存分配触发：当堆内存达到上次 GC 后的 2 倍时触发（GOGC 控制）
  2. 定时触发：2 分钟未触发则强制触发
  3. 手动触发：调用 runtime.GC()
import "runtime"

// 手动触发 GC（一般不需要）
runtime.GC()

// 查看 GC 统计
var m runtime.MemStats
runtime.ReadMemStats(&m)
fmt.Printf("GC 次数: %d\n", m.NumGC)
fmt.Printf("堆内存: %d MB\n", m.HeapAlloc/1024/1024)

GC 调优

GOGC 环境变量

控制 GC 触发阈值，默认值为 100（堆内存增长 100% 触发）：

# 减少 GC 频率（适合内存充足场景）
export GOGC=200

# 增加 GC 频率（适合低延迟场景）
export GOGC=50

# 禁用 GC（仅调试使用）
export GOGC=off

GOMEMLIMIT

Go 1.19+ 引入软内存限制，控制 Go 程序内存使用：

export GOMEMLIMIT=4GiB

优化建议

问题解决方案
GC 频繁减少临时对象分配，使用 sync.Pool
内存占用高检查内存泄漏，及时释放大对象引用
GC 延迟高降低 GOGC，减少堆大小

GC 监控

// 打印 GC 日志
import "os"
os.Setenv("GODEBUG", "gctrace=1")

// 输出示例：
// gc 1 @0.005s 0%: 0.015+0.23+0.012 ms clock, 0.12+0.18/0.33/0.045+0.098 ms cpu, 4->4->0 MB, 5 MB goal, 8 P

日志格式说明：

  • gc 1：第 1 次 GC
  • @0.005s：程序启动后 0.005 秒
  • 0.015+0.23+0.012 ms：STW 标记 + 并发标记 + STW 清除 的耗时
  • 4->4->0 MB：GC 前堆大小 -> GC 后堆大小 -> 存活大小

总结

特性说明
算法并发三色标记-清除
STWGo 1.8+ 通常 < 100μs
触发条件堆增长达到 GOGC% 或 2 分钟
调优手段GOGC、GOMEMLIMIT、减少分配

Go GC 设计目标是低延迟，适合对响应时间敏感的服务。通过减少堆内存分配、复用对象，可以进一步降低 GC 压力。
