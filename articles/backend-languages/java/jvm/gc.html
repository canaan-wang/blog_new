Java 垃圾回收（GC）

一、GC 核心概念

垃圾回收（Garbage Collection）是 JVM 自动管理内存的机制，负责回收不再使用的对象，防止内存泄漏。

判断对象死亡

  1. 引用计数法：计数为 0 时回收（Python 用，Java 不用，无法解决循环引用）
  2. 可达性分析：从 GC Roots 向下搜索，不可达的对象可回收
// GC Roots 包括：
// 1. 虚拟机栈中局部变量引用的对象
// 2. 方法区中类静态属性引用的对象
// 3. 方法区中常量引用的对象
// 4. 本地方法栈中 JNI 引用的对象
// 5. 所有被同步锁持有的对象

二、垃圾回收算法

算法原理优点缺点
标记-清除标记存活对象，清除未标记简单内存碎片
复制存活对象复制到另一块内存无碎片内存利用率 50%
标记-整理标记后移动对象，清理边界外无碎片移动对象开销大
分代收集年轻代用复制，老年代用标记-清除/整理高效实用复杂

分代策略

  • 年轻代：存放新对象，Minor GC 频繁，大部分对象很快死亡
  • 老年代：存放长期存活对象，Major GC 较少
  • 晋升：对象存活足够次数（默认 15）后进入老年代

四、垃圾回收器

回收器算法特点适用场景
Serial复制/标记整理单线程，STW客户端模式
ParNew复制Serial 多线程版配合 CMS
Parallel Scavenge复制吞吐量优先后台计算
CMS标记清除低延迟，并发Web 应用（JDK 9 废弃）
G1标记整理 + 复制分区管理，可预测停顿大堆（JDK 9+ 默认）
ZGC染色指针超低延迟（<10ms），并发JDK 11+，超大堆
Shenandoah染色指针低延迟，并发OpenJDK，超大堆

五、G1 垃圾回收器详解

G1（Garbage First）是 JDK 9+ 默认回收器，适合大堆（> 6GB）：

G1 特点

  • 堆划分为多个 Region（1MB~32MB）
  • 优先回收垃圾最多的 Region
  • 可设置目标停顿时间（默认 200ms）
  • 整体标记整理，局部复制

六、GC 调优

常用参数

# 基础配置
-Xms4g -Xmx4g          # 固定堆大小，避免扩容
-XX:+UseG1GC           # 使用 G1
-XX:MaxGCPauseMillis=200  # 目标停顿时间

# G1 高级配置
-XX:G1HeapRegionSize=16m   # Region 大小
-XX:InitiatingHeapOccupancyPercent=45  # 触发 GC 的堆占用率

# GC 日志（JDK 9+）
-Xlog:gc*:file=gc.log:time,uptime,level,tags

调优目标选择

目标策略
低延迟选择 CMS/G1/ZGC，减小堆，缩短停顿时间目标
高吞吐选择 Parallel/G1，增大年轻代，减少 GC 次数
大堆必须使用 G1/ZGC/Shenandoah

七、GC 问题排查

常见 GC 问题

问题现象解决
内存泄漏Full GC 频繁，内存不释放heap dump 分析，MAT 工具
GC 频繁CPU 高，吞吐量低增大堆，减少对象创建
Full GC 时间长应用卡顿改用 G1/ZGC，减小堆
OOM堆/元空间/直接内存溢出分析 dump 文件，修复代码

查看 GC 情况

# 实时查看 GC
jstat -gcutil PID 1000

# 输出解释

0.00  80.00  20.00  30.00  95.00  92.00     10    0.150     2    0.500    0.650

# S0/S1: Survivor 使用率
# E: Eden 使用率
# O: Old 使用率
# M: Metaspace 使用率
# YGC/YGCT: 年轻代 GC 次数/时间
# FGC/FGCT: Full GC 次数/时间

八、最佳实践

  • 避免创建大对象（直接进入老年代）
  • 及时释放不再使用的对象引用（help GC）
  • 使用连接池、对象池减少对象创建
  • 设置合理的堆大小，避免频繁扩容
  • 生产环境开启 GC 日志，便于问题排查
  • 定期分析 GC 日志，调整参数
